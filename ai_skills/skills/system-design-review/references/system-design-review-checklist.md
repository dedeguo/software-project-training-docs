# 系统设计专项评审清单

仅在全面评审或相关专项评审时读取。按项目范围选择检查项，不相关项标记“不适用”，不得为了满足清单而新增功能或复杂技术。

## 1. 评审材料与追踪

- 立项、需求、设计、原型、数据库和接口材料版本明确且相互匹配。
- 图片、链接、附件和脚本可访问，图中文字清楚。
- 所有待确认事项集中列出，没有伪装成确定设计。
- 每个必做需求能追踪到模块、页面或调用方、接口、权限/状态、数据和验证方式。
- 每个设计元素都有需求、规则或技术约束依据。
- 术语、编号、字段、状态值和版本在图、表、正文中一致。

## 2. 技术选型、架构与模块

- 技术方案解决明确问题，符合团队能力、课程周期和目标环境。
- 技术及版本相互兼容，安装、调试和部署条件明确。
- 外部 API、服务、费用和限制已确认，并有必要的失败或降级方案。
- 架构图中的组件真实存在、职责清楚、调用方向合理。
- 开发架构与部署架构一致，一次核心请求能够解释。
- 模块按业务职责划分，边界清楚，没有重复职责或循环依赖。
- 微服务、消息队列、缓存等复杂设施有真实依据和验证方法。

## 3. 页面、原型与用户流程

- 页面清单覆盖核心角色和任务，没有无需求依据的页面功能。
- 用户完成任务所需信息齐全，每个展示字段有可靠数据来源。
- 主要操作、危险操作、确认、取消和返回路径明确。
- 表单必填、格式、边界和错误位置明确。
- 加载、空数据、失败、无权限和业务状态受限中实际需要的状态已经设计。
- 处理中能防止重复提交，按钮状态与后端权限和业务状态一致。
- 登录失效、直接访问受限页面和操作失败后有恢复路径。

## 4. 数据模型与数据库

- 核心业务对象、属性和关系来自真实需求，不把页面控件设计成表。
- 一对一、一对多、多对多关系和基数正确；中间表保留必要业务属性。
- 每张表用途、主键、关联关系、字段类型、长度、可空性和默认值明确。
- 状态、时间、金额、密码、删除、历史和审计字段处理合理。
- 主键、外键、非空、唯一和检查约束支撑业务规则。
- 索引来自真实查询、关联和唯一性需求，没有机械给所有字段加索引。
- 页面字段和接口字段能找到数据来源，类型和取值一致。
- 建表脚本能在目标数据库空库执行，初始化数据能支持核心流程。
- 重复账号、无效关联、非法状态等错误数据受到阻止。

## 5. 接口与数据一致性

- 每个核心页面操作或外部调用有对应接口，多余接口有明确用途。
- 方法、路径、模块归属、分页、筛选、时间和状态约定统一。
- 路径参数、查询参数、请求体、类型、必填和校验规则明确。
- 响应结构支持页面需要，主要成功和失败情况有稳定结果。
- HTTP 状态与业务错误码能区分未登录、无权限、校验和业务失败。
- 改变数据的接口说明前置条件、数据变化和失败后状态。
- 多表更新和状态变化具有明确事务边界，不留下部分成功数据。
- 重复提交、连续点击和重试不会产生重复业务结果。
- 最后一个库存、名额等并发竞争有服务层和数据库层保护。

## 6. 认证、权限、安全与隐私

- 登录凭证传递、失效和退出流程明确；未登录与无权限结果不同。
- 密码使用可靠单向哈希，不使用明文或可逆加密。
- 每项管理功能在后端校验角色，不只依赖菜单或按钮隐藏。
- “我的数据”从当前登录身份确定用户范围，不信任客户端用户编号。
- 普通用户不能通过修改路径或请求参数访问他人数据。
- 查询和导出在后端或数据库层限制数据范围。
- 只收集业务必要数据，文档、日志和测试数据不含真实敏感信息。
- 密钥、密码和本机配置不会进入仓库；错误响应不泄露堆栈或数据库细节。
- 上传、删除、导出、日志和第三方服务按项目实际风险采取保护。

## 7. 业务状态、异常与边界

- 每个状态具有唯一含义、稳定取值、初始状态和结束状态。
- 所有状态都能合法进入和处理，不存在无出口或任意跳转。
- 每次转换明确操作角色、前置条件、修改数据、下一状态和失败结果。
- 页面按钮、接口动作、数据库状态和状态机一致。
- 状态变化和关联写入需要原子性时处于同一事务。
- 重复操作、非法转换、数据不存在、库存/名额不足等有明确拒绝结果。
- 并发操作和外部服务超时不会破坏核心数据正确性。
- 必要的操作时间、操作者和状态历史能够追踪。

## 8. 部署、质量与可验证性

- 开发、测试和演示/生产环境及其差异明确。
- 部署结构、运行版本、依赖服务和配置方式可执行。
- 环境变量、密钥、初始化、备份、恢复、日志和必要监控有安排。
- 性能、容量、安全、可用性等目标有需求依据和可观察标准。
- 高风险技术点有最小验证计划和完成证据。
- 权限矩阵可转化为允许/拒绝测试，状态机可转化为状态测试。
- 初始化数据支持成功、失败、越权和边界场景。
- 未实际运行的 SQL、接口、部署和并发方案明确标记为未验证。

## 9. 核心流程最低走查集

按实际业务替换对象，至少检查：

| 场景 | 要确认的结果 |
| --- | --- |
| 正常完成核心任务 | 页面、接口、状态和数据都得到预期结果 |
| 未登录访问 | 接口拒绝，页面引导登录，数据不变 |
| 角色错误 | 后端拒绝，不依赖前端按钮隐藏 |
| 修改用户或对象编号 | 不能越权访问或操作他人数据 |
| 当前状态不允许 | 返回稳定业务错误，状态和数据不变 |
| 重复提交或连续点击 | 不产生重复记录、重复扣减或重复状态变化 |
| 两个请求竞争最后资源 | 只有合法数量成功，约束保持成立 |
| 多表写入中途失败 | 整体回滚，不留下部分成功数据 |
| 外部服务失败 | 错误可解释，核心数据不被破坏，必要时降级 |

## 10. 进入设计基线的门槛

- 核心需求均有可实现设计支撑。
- 核心成功、失败和权限路径能够完整走通。
- 阻断问题全部关闭。
- 重要问题关闭，或有证据表明不影响当前开发并具有明确计划。
- 架构、页面、数据库、接口、权限和状态基本一致。
- 关键脚本和环境已经验证，未验证项的影响明确。
- 待确认事项不阻塞即将开始的核心开发。
- 评审对象、版本、结论、未关闭问题和确认人员记录完整。
