# 课程项目选题库（topic-selection 资源）

> 本文件是 `topic-selection` Skill 的参考选题库，用于比较题目、校准难度和控制范围，不是唯一允许选择的题目清单。
>
> 课程原则：AI 辅助能够提高开发效率，**默认推荐学生一人一题、独立完成**。自拟题目和组队项目须经任课教师审核后再进入开发阶段。

---

## 选题判断标准

一个适合课程实训的题目应当具备：

1. **明确用户**：具体说明谁使用，避免“所有人”；
2. **真实问题**：能说明用户当前怎样做、哪里不方便；
3. **业务闭环**：至少有一条从开始到结果的完整流程；
4. **范围可控**：核心版本适合个人在课程周期内完成；
5. **条件可得**：数据、接口、软件和硬件基本可获得；
6. **可以验收**：能够运行、测试、演示，并说明个人完成内容。

出现下列情况时，应更换题目或重新缩小范围：没有具体用户、没有业务闭环、关键数据或服务不可获得、主体只是调用外部 API、明显超出课程周期，或涉及难以控制的支付、隐私、安全与合规风险。

## 难度与完成方式

| 难度 | 适合情况 | 建议范围 | 完成方式 |
| --- | --- | --- | --- |
| **L1 基础** | 第一次完成 Web 项目或基础较弱 | 2～3 类用户、1 条主流程、基础权限和管理功能 | 1 人独立完成 |
| **L2 标准** | 已做过简单增删改查 | 多角色、状态流转、1～2 条主流程、审核或统计 | 1 人独立完成 |
| **L3 进阶** | Web 基础较熟练，希望增加特色能力 | 完整业务系统＋1 项 AI、推荐或复杂特色功能 | 优先 1 人；确需组队须审核 |

建议多数学生选择 **L1 或 L2**。L3 不等于必须组队，应先缩小范围并评估能否独立完成。

确需 2～3 人组队时，项目必须具有真实的工作量和清晰边界；每名成员须承担可独立演示、测试和答辩的核心模块，并通过 Git 提交、任务记录和阶段成果证明个人贡献。

## 范围控制原则

统一将功能分为四类：

| 优先级 | 含义 | 示例 |
| --- | --- | --- |
| 必做 | 没有它就无法形成核心闭环 | 报修中的提交、派单、处理和确认 |
| 应做 | 保证系统基本完整 | 查询筛选、状态记录、基础权限、简单统计 |
| 选做 | 核心版本完成后再增加 | 智能推荐、消息提醒、数据可视化 |
| 本期不做 | 主动排除以避免失控 | 在线支付、复杂即时通信、跨校数据共享 |

个人项目通常控制在：

- 2～3 类用户；
- 4～6 个核心模块；
- 1～2 条主要业务流程；
- 最多 1 项 AI 或特色增强功能。

如果一句话无法讲清主流程，或个人无法在答辩中完整演示，通常说明范围仍需缩小。

## 自拟题目与组队审核

学生可以提出选题库外的项目，选题库只作为范围标尺。审核流程：

1. 使用 `topic-selection` Skill 检查用户、问题、闭环、数据、技术和范围；
2. 完成《项目选题立项书》与四级功能清单；
3. 自拟题目经任课教师审核后进入后续阶段；
4. 组队项目另行说明组队必要性、成员职责、个人验收成果和贡献留痕方式；
5. 无法清楚评价个人贡献的组队方案，应缩小为独立项目或重新分工。

## 推荐策略

- 没有明确想法：推荐 2～3 个适合其基础且能独立完成的题目；
- 有候选题目：评分比较后给出有条件的明确推荐；
- 有创意题目：优先保留创意，再参考相似项目校准难度；
- 题目过大：先缩小为 L1/L2 个人版本；
- 希望组队：先判断是否能缩小为个人项目，确有必要再提交教师审核；
- AI 题目：先验收普通业务闭环，再增加 AI 能力。
