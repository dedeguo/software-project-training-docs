# 系统设计一致性检查清单

在评审完整设计、补全文档或准备进入编码前读取本清单。根据项目实际范围检查，不机械要求不适用的内容。

## 1. 需求与范围

- 每个必做需求都有对应模块和实现路径。
- 每个模块、页面、接口和数据表都有明确需求依据。
- 选做内容与必做内容已经区分。
- 本期不做内容没有悄悄进入设计。
- 待确认需求没有被写成确定设计。

## 2. 技术方案与架构

- 技术选型符合团队能力、课程周期、运行和部署环境。
- 关键方案记录了候选项、决定、依据、风险和影响。
- 架构图中的每个组件都有职责，调用方向明确。
- 核心请求能够从页面追踪到后端、数据库和响应。
- 模块职责不重叠，边界清楚，没有明显循环依赖。
- 中间件和复杂架构都有真实需求依据和验证方法。

## 3. 页面与交互

- 核心场景能够通过页面流程完整走通。
- 每个关键操作都对应接口、权限和允许状态。
- 页面展示字段能够从接口响应获得。
- 成功、空、加载、失败、无权限和状态受限中需要的页面状态已经说明。
- 页面按钮、提示和状态名称与接口及状态机一致。

## 4. 数据库

- 实体、关系和字段来源于业务对象及规则，不仅来源于页面表单。
- 主键、外键、唯一、非空和检查约束支撑核心业务规则。
- 一对一、一对多、多对多关系实现合理。
- 时间、金额、状态、删除和审计字段约定统一。
- 高频查询、关联和唯一性需求有适当索引。
- 重复提交、并发竞争和历史数据处理方式明确。
- 敏感数据遵循最小化原则，没有明文密码或真实密钥。

## 5. 接口

- 接口方法、路径和资源名称一致且语义明确。
- 请求字段与页面输入、数据库字段和业务规则一致。
- 响应字段能够支持页面展示和下一步操作。
- 必填、格式、范围、对象存在、权限和状态校验明确。
- 成功响应、主要失败响应和错误码明确。
- 修改多个相关数据或状态时，事务边界明确。
- 重复请求或并发请求可能造成错误时，幂等或并发策略明确。

## 6. 权限与状态

- 功能权限和数据权限已经分别说明。
- 后端执行最终权限校验，不依赖前端隐藏按钮。
- 普通用户只能访问本人或明确授权范围的数据。
- 每个状态有明确含义和稳定取值。
- 每次转换都有当前状态、操作、角色、条件、下一状态和失败结果。
- 非法跳转、完成后重复操作、取消后继续处理等情况受到限制。
- 页面、接口、数据库使用相同状态名称或有明确映射。

## 7. 部署、质量与安全

- 部署图与实际采用的组件和外部服务一致。
- 环境变量和密钥不写入代码或正式文档。
- 初始化、备份、恢复和必要日志方案能够执行。
- 性能或容量指标有需求依据、假设和验证方法。
- 外部服务失败时有明确错误处理或降级方案。
- 上传、认证、输入校验和敏感数据保护覆盖项目实际风险。

## 8. 追踪与可验证性

- `FR/BR → MOD → UI → API → ROLE/ST → TABLE → AC/测试` 可以双向追踪。
- 图、表、接口、页面和决定使用稳定编号并在正文引用。
- 建表 SQL、接口示例和部署步骤标明已验证或未验证。
- 所有 `【待学生确认】` 已集中列出，说明影响和负责人。
- 一个未参与设计的人能够只依靠文档解释并实现核心闭环。

## 问题输出格式

| 级别 | 位置/编号 | 问题 | 影响 | 需求依据 | 最小修改建议 |
| --- | --- | --- | --- | --- | --- |
| 必须修改 / 待确认 / 可选优化 | 【填写】 | 【具体问题】 | 【会导致什么后果】 | FR-* / BR-* / 无 | 【最小修改方式】 |

使用以下分级：

- **必须修改**：核心闭环断裂、需求冲突、越权、状态矛盾、数据错误、方案不可实现或无法验证。
- **待确认**：材料不足，且不同答案会改变设计。
- **可选优化**：不影响本期核心实现，可提高表达、维护性或体验。

不要把新增功能或技术升级包装成“必须修改”。
