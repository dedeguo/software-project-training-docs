# 使用AI完成功能迭代

本节在已经运行的示例项目上完成一次小型功能迭代，体验“明确任务、AI提出方案、人工审核、编码测试、Git提交”的完整过程。

## 学习目标

- 能将想法改写为可验证的开发任务；
- 能向 AI 提供项目上下文和修改约束；
- 能审核 AI 给出的方案和代码；
- 能通过运行和测试验证修改结果；
- 能用 Git 记录一次完整迭代。

## 选择一个小任务

任务应在一次实践课内完成，例如：

- 为图书列表增加书名或作者关键字查询；
- 为借阅记录增加状态筛选；
- 补充必填项校验和错误提示；
- 修复一个可以稳定复现的问题。

不要选择重写系统、替换技术栈或一次增加多个业务模块等过大任务。

## 编写任务卡

```markdown
# 任务名称
## 背景与目标
## 当前项目与技术栈
## 涉及的用户和业务场景
## 功能要求
## 不允许修改的范围
## 验收标准
## 测试方法
```

验收标准要能够直接检查。例如，“支持按书名模糊查询”比“优化搜索功能”更容易验证。

## 与 AI 协作

```text
请先阅读当前项目结构和与【任务名称】有关的代码，不要立即修改。

任务目标：【填写】
技术环境：【填写】
已有规范：【填写】
范围限制：【填写】
验收标准：【填写】

请先给出：
1. 对现有实现的理解；
2. 需要修改的文件及原因；
3. 分步实施计划；
4. 可能的风险和测试方法。
```

人工确认方案后，再让 AI 分步骤修改。每完成一步都应检查差异，避免覆盖无关代码。

## 运行与测试

至少验证：

1. 正常输入是否得到预期结果；
2. 空值、错误值和边界值是否被正确处理；
3. 原有功能是否仍然可用；
4. 前端提示、接口返回和数据库结果是否一致；
5. 构建和启动过程是否成功。

记录测试输入、预期结果、实际结果和问题修正过程。不能只用“运行成功”代替测试证据。

## 完成 Git 提交

提交前检查：

- 只包含本次任务相关修改；
- 没有密码、密钥、数据库备份和构建产物；
- 代码能够构建并通过测试；
- 提交信息能够说明本次变化。

示例：

```text
feat: 增加图书关键字查询
```

## 提交成果

- 开发任务卡；
- AI协作记录；
- 修改前后对比；
- 测试记录；
- Git提交记录；
- 可运行的功能演示。

## 验收检查

- [ ] 任务范围小而明确；
- [ ] AI修改前先分析并提出计划；
- [ ] 学生能够解释关键修改；
- [ ] 验收标准均有测试证据；
- [ ] Git提交与实际修改一致。
