# 项目选题与范围控制

## 好选题的四个标准

1. 有明确用户，而不是“所有人”；
2. 有真实业务流程，而不是若干数据表的增删改查；
3. 16周内能够完成核心版本；
4. 具备一项可展示的特色功能。

## 推荐选题结构

```text
面向【目标用户】
解决【具体问题】
通过【核心业务流程】
形成【可交付的软件系统】
```

示例：

> 面向高校竞赛指导教师和学生，解决竞赛通知分散、报名材料混乱和过程跟踪困难的问题，建设包含赛事发布、团队报名、材料审核、进度跟踪和成果统计的竞赛项目管理系统。

## 范围控制

将功能划分为：

- 必做：没有它就无法形成业务闭环；
- 应做：提高项目完整性；
- 选做：时间充足时实现；
- 暂不实现：明确排除，防止项目失控。

## 阶段验收

- 项目名称清楚；
- 目标用户明确；
- 至少描述一个真实使用场景；
- 必做功能可以形成业务闭环；
- 没有把“大数据、区块链、AI”等词语简单堆砌。
