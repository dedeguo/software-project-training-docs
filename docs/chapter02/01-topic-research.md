# 项目选题、调研与范围控制

项目选题不是先决定使用什么技术，而是先找到一个值得解决、能够验证并能在课程周期内完成的问题。

## 好选题的四个标准

1. 有明确用户，而不是面向“所有人”；
2. 有真实业务流程，而不是若干数据表的简单增删改查；
3. 能在 16 周内完成一个小而完整的核心版本；
4. 有可运行、可测试、可展示的成果。

## 描述项目想法

使用下面的句式形成项目的一句话介绍：

```text
面向【目标用户】，解决【具体问题】，
通过【核心业务流程】，形成【可交付的软件系统】。
```

示例：

> 面向高校竞赛指导教师和学生，解决竞赛通知分散、报名材料混乱和过程跟踪困难的问题，通过赛事发布、团队报名、材料审核和进度跟踪，形成竞赛项目管理系统。

## 开展初步调研

至少选择两种调研方式：

- 访谈或观察目标用户；
- 体验 2～3 个同类产品；
- 查阅现有工作表、表单或管理制度；
- 收集典型问题、操作记录或业务材料。

调研时重点记录：用户当前怎么做、哪里最耗时、哪些错误最常见、现有工具为什么不能满足需要。

## 分析项目价值

| 分析角度 | 需要回答的问题 |
| --- | --- |
| 用户价值 | 项目能帮助谁减少什么困难？ |
| 业务价值 | 能缩短时间、减少错误还是改善协作？ |
| 技术可行性 | 关键数据、接口和技术条件是否具备？ |
| 展示价值 | 能否用一条完整流程说明项目成果？ |

不要用“结合 AI、大数据、区块链”等技术名词代替价值分析。

## 控制项目范围

将功能划分为：

- **必做：** 没有它就无法形成核心业务闭环；
- **应做：** 能提高项目完整性，但不影响最小版本运行；
- **选做：** 时间充足时再实现的增强功能；
- **暂不实现：** 本学期明确排除的内容。

先画出一条最小业务闭环，再决定需要哪些功能。原则上先完成小而完整的项目，再增加特色能力。

## 提交成果

- 项目一句话介绍；
- 用户调研或竞品分析记录；
- 项目价值分析；
- 最小业务闭环；
- 必做、应做、选做和暂不实现功能清单。

## 验收检查

- [ ] 用户和问题具体明确；
- [ ] 价值判断具有调研依据；
- [ ] 必做功能能够形成业务闭环；
- [ ] 项目范围适合在课程周期内完成；
- [ ] 特色功能与项目业务有关。
