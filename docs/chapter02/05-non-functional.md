# 非功能需求与验收标准

非功能需求描述系统在性能、安全、易用性和可靠性等方面应达到什么水平。它们必须具体、可测量，不能只写“响应快”“安全性高”或“界面友好”。

## 常见非功能需求

| 类别 | 模糊表述 | 可验收表述示例 |
| --- | --- | --- |
| 性能 | 系统响应快速 | 在课程验收数据量下，主要查询 95% 在 2 秒内返回 |
| 安全 | 系统安全可靠 | 密码加密存储，未登录和无权限请求由后端拒绝 |
| 易用性 | 页面简单易用 | 表单显示必填项，校验失败时指出具体字段和原因 |
| 可靠性 | 系统稳定运行 | 业务异常不导致服务退出，并记录可定位的错误日志 |
| 兼容性 | 兼容常用浏览器 | 在指定版本的 Chrome 和 Edge 中完成核心流程 |
| 可维护性 | 代码结构清晰 | 统一返回、异常和日志规范，无密码或密钥进入仓库 |

指标应结合课程项目规模确定，不要照搬大型互联网系统的并发量和可用性指标。

## 编写验收标准

推荐使用“给定—当—那么”描述：

```text
给定：普通用户已经登录；
当：该用户直接访问管理员接口；
那么：后端返回无权限结果，且数据不发生变化。
```

每条标准应说明：

- 测试前提；
- 操作或输入；
- 可观察结果；
- 必要的数据或环境；
- 对应的需求编号。

## 覆盖异常与边界

除正常流程外，至少考虑：

- 必填值为空；
- 长度、数量和日期达到边界；
- 重复提交；
- 未登录和越权访问；
- 数据不存在或状态不允许；
- 第三方服务不可用；
- 网络超时或数据库操作失败。

## 建立需求与验收对应关系

| 需求编号 | 验收场景 | 测试方式 | 通过条件 |
| --- | --- | --- | --- |
| FR-TEAM-01 | 正常创建团队 | 页面与接口测试 | 团队记录创建且状态正确 |
| BR-TEAM-02 | 成员人数超限 | 接口边界测试 | 请求被拒绝并返回明确原因 |
| NFR-SEC-01 | 普通用户越权 | 权限测试 | 后端拒绝且数据不变 |

## 提交成果

- 非功能需求清单；
- 核心需求验收标准；
- 异常与边界场景清单；
- 需求—验收对应表。

## 验收检查

- [ ] 非功能需求适合项目实际规模；
- [ ] 每项要求都可以通过测试或检查验证；
- [ ] 权限、安全和隐私要求没有遗漏；
- [ ] 核心需求覆盖正常、异常和边界场景；
- [ ] 验收标准没有“良好、合理、尽量”等模糊词语。
