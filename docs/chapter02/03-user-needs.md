# 用户需求与使用场景

需求分析首先要理解用户，而不是直接列功能菜单。本节通过用户角色、使用场景和用户故事说明系统为什么需要某项功能。

## 识别用户角色

对每类角色回答：

- 他是谁，承担什么职责？
- 在什么情况下使用系统？
- 想完成什么任务？
- 当前方式有什么困难？
- 完成任务后如何判断成功？

角色应根据目标和权限区分，不要因为部门或姓名不同就创建新的角色。

| 用户角色 | 主要目标 | 典型任务 | 关注点 |
| --- | --- | --- | --- |
| 学生 | 完成竞赛报名 | 查找赛事、组队、提交材料 | 截止时间、材料状态 |
| 指导教师 | 指导并审核材料 | 查看团队、提出修改意见 | 材料完整性、进度 |
| 竞赛负责人 | 管理赛事过程 | 发布赛事、审核、统计 | 规则执行、数据准确性 |

## 描述使用场景

一个场景至少包括：

1. 场景名称；
2. 参与角色；
3. 触发条件；
4. 用户目标；
5. 当前做法和困难；
6. 期望结果；
7. 异常或限制。

示例：学生在报名截止前创建团队并上传材料；如果成员未确认或材料缺失，系统应提示具体问题，不能直接提交。

## 编写用户故事

```text
作为【某类用户】，
我希望【完成某项任务】，
以便【获得某种价值】。
```

示例：

> 作为竞赛负责人，我希望审核学生提交的报名材料，以便在截止日期前确认材料是否完整。

用户故事还需要补充验收条件，例如：能够查看待审核记录、给出通过或退回结果，并记录审核意见和时间。

## 排定需求优先级

- **必须有：** 支撑核心业务闭环；
- **应该有：** 明显改善完整性和效率；
- **可以有：** 有价值但可以延后；
- **本期不做：** 明确排除。

优先级必须说明依据，不能把所有需求都标记为“必须有”。

## 提交成果

- 用户角色表；
- 3～5个典型使用场景；
- 核心用户故事及验收条件；
- 需求优先级清单；
- 调研证据或访谈摘要。

## 验收检查

- [ ] 每个角色都有明确目标和职责；
- [ ] 场景来自真实调研或合理业务依据；
- [ ] 用户故事说明了用户价值；
- [ ] 核心角色和异常场景没有明显遗漏；
- [ ] 优先级与MVP范围一致。
