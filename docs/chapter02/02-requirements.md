# 用户需求与业务流程

## 从用户角色开始

先回答：

- 谁使用系统？
- 他在什么情况下使用？
- 想完成什么任务？
- 当前方式有什么困难？
- 完成后如何判断成功？

## 用户故事格式

```text
作为【某类用户】，
我希望【完成某项任务】，
以便【获得某种价值】。
```

示例：

> 作为竞赛负责人，我希望审核学生提交的报名材料，以便在截止日期前确认材料是否完整。

## 功能清单不是菜单清单

不推荐：

```text
用户管理、赛事管理、报名管理、材料管理。
```

推荐：

```text
学生查看可报名赛事 → 创建团队 → 邀请成员 → 提交报名材料
→ 指导教师审核 → 学生修改 → 审核通过 → 进入项目跟踪。
```

## AI审查提示词

```text
请以软件需求评审人员的身份审查以下需求。
重点检查：
1. 用户角色是否遗漏；
2. 主流程能否闭环；
3. 异常和边界场景是否缺失；
4. 功能是否超出16周项目范围；
5. 只提出问题和建议，不替我虚构需求。
```

## 阶段提交

- 用户角色表；
- 用户故事；
- 功能清单；
- 一张核心业务流程图；
- 必做、选做和暂不实现功能；
- AI审查记录及人工修改说明。
