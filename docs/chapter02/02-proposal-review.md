# 项目申报、计划与立项评审

确定选题后，需要把调研结论、项目范围和实施计划整理为可评审的项目申报材料。评审的目的不是追求一次通过，而是尽早发现范围、价值和可行性问题。

## 项目申报内容

项目计划书至少包括：

1. 项目名称；
2. 项目背景和问题描述；
3. 目标用户与使用场景；
4. 调研过程和主要发现；
5. 项目目标与价值；
6. 核心业务流程；
7. MVP范围和功能优先级；
8. 初步技术方案；
9. 16周里程碑计划；
10. 团队分工、风险和应对措施。

## 制定里程碑

里程碑要描述可以检查的成果，而不是“学习框架”“继续开发”等过程。

| 阶段 | 可检查成果 |
| --- | --- |
| 选题与设计 | 立项书、需求文档、原型和设计方案 |
| 项目骨架 | 可启动工程、登录权限和第一个核心模块 |
| 业务开发 | 完整业务闭环和特色功能 |
| 测试与发布 | 测试报告、部署地址和演示材料 |
| 最终验收 | 最终作品、答辩材料和个人总结 |

## 分析风险

至少检查：

- 项目范围是否过大；
- 关键数据或第三方服务是否可获得；
- 团队成员是否具备必要基础；
- AI或特色功能失败时是否有降级方案；
- 部署、测试和成果整理是否预留时间。

## 进行立项评审

评审人员重点检查：

1. 用户和问题是否真实；
2. 调研结论是否支持项目价值；
3. MVP是否能够形成完整业务流程；
4. 功能数量和技术难度是否可控；
5. 成果是否可运行、可测试、可展示；
6. 团队分工和里程碑是否合理。

评审结论可分为：通过、修改后通过、重新选题。所有意见都应记录“是否采纳、修改内容和理由”。

## AI辅助审查

```text
请以课程项目立项评审人员的身份审查以下计划书。
重点检查用户与问题是否明确、调研证据是否充分、
MVP能否形成闭环、16周范围是否可控、风险是否遗漏。
请指出具体问题和修改建议，不要替我虚构调研结论。
```

## 提交成果

- 项目计划书；
- 16周里程碑计划；
- 风险清单；
- 立项评审记录；
- 修改说明和通过版本。

## 验收检查

- [ ] 项目目标与调研结果一致；
- [ ] MVP边界清楚；
- [ ] 里程碑均有可检查成果；
- [ ] 风险具有具体应对措施；
- [ ] 评审意见已逐条处理。
