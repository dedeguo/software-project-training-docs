# 1.3 建立 AI 规则：Trae 规则、Skills 与 AGENTS.md

在 1.2 中，我们认识了 Trae IDE 与 Superpowers-zh。但工具再强，也得有规矩约束它别“自由发挥”——否则它可能跳过测试、擅改表结构、生成游离于设计文档之外的接口。

本节正式介绍 Trae 管控 AI 行为的三大基础设施：**规则（Rules）、技能（Skills）、AGENTS.md**，并顺带说明常用的 **MCP 工具**。读完这一节，你就知道为什么“给 AI 立规矩”要分成这三层来做。

## 一、先搞懂：Rules、Skills、MCP 是什么关系

很多同学刚上手 Trae 会混淆这三个概念。官方社区有一个很贴切的比喻：

> **MCP 是锤子，Skills 是钉钉子的技能，Rules 是钉钉子的施工规范——不允许随便乱锤。**

| 维度 | Rules（规则） | Skills（技能） | MCP（工具） |
|---|---|---|---|
| 本质 | 底线约束 | 知识规范 | 工具能力 |
| 类比 | 施工安全守则 | 木工手册 | 锤子、扳手 |
| 加载机制 | **全量加载**，对话开始即注入上下文 | **按需加载**，仅相关时才注入 | 按需调用 |
| 占用上下文 | 持续占用 | 用到才占 | 不占上下文，占工具配额 |
| 适合放什么 | 项目红线、技术栈约束 | 复杂工作流方法论（TDD、代码审查） | 浏览器、文档检索、记忆等外部能力 |
| 数量建议 | 精简，单条 ≤ 1000 字符 | 可按需创建 | 够用就好，工具总数上限 40 |

**核心策略：用最少的 MCP 满足核心需求，把方法论放进 Skills，用 Rules 守住底线。**

理解了三者的分工，下面逐个展开。

## 二、Trae 的规则体系（`.trae/rules/`）

### 1. 规则放在哪、分几种

Trae 用 `.trae/rules/` 目录管理 AI 行为规则，规则以 Markdown 编写，带 frontmatter 元数据（`description`、`globs`、`alwaysApply`、`priority`）。

按作用域，规则分两类：

| 规则类型 | 文件位置 | 生效范围 | 适合放 |
|---|---|---|---|
| **项目规则** | 项目内 `.trae/rules/` | 仅当前项目（提交 Git，团队共享） | 技术栈约束、架构规范、安全要求 |
| **个人规则** | 用户主目录 `~/.trae/` | 所有项目 | 个人语言偏好、代码风格、交互习惯 |

> **⚠️ 优先级：** 当个人规则与项目规则冲突时，**项目规则优先级更高**，不会被个人偏好覆盖。即 `项目 rules > 用户 rules > 默认行为`。

### 2. 四种生效方式

创建项目规则时，需要指定它的生效方式（系统会自动维护对应的 `alwaysApply` 等字段）：

| 生效方式 | 描述 | 典型场景 |
|---|---|---|
| **始终生效** | 当前项目所有对话都注入 | 项目红线、技术栈约束 |
| **指定文件生效** | 仅当对话中提及的文件匹配 `globs` 通配符时注入 | `src/server/api/**/*.ts` 下的接口校验规则 |
| **智能生效** | 由 AI 根据 `description` 判断相关性后决定是否使用 | “编写 React 组件测试时使用” |
| **手动触发生效** | 仅当用 `#Rule` 提及某规则时才生效 | 偶尔用到的特殊规范 |

### 3. superpowers-zh 的 bootstrap 规则

执行 `npx superpowers-zh` 安装时，除了把 skills 复制到 `.trae/skills/`，还会在 `.trae/rules/superpowers-zh.md` 生成一份 **bootstrap 规则**。它的作用是：

- 在对话开始时被全量注入（始终生效）；
- 引导 AI 在任务匹配时去读取 `.trae/skills/<skill-name>/SKILL.md` 并遵循其流程；
- 让 skills 不必常驻上下文，又能在合适时机自动触发。

你可以把它理解成 superpowers 的“启动器”——没有它，手动 `cp -r skills` 只是把文件放进去，AI 不会在合适时机主动调用，需要你每次手动喊“use brainstorming skill”。

## 三、Skill 的概念（`.trae/skills/`）

### 1. Skill 是什么

在 Trae 中，**技能（Skill）通过 `SKILL.md` 文件定义和管理**。每个技能封装了指令、脚本及相关资源，为智能体提供可复用、面向特定场景的专业能力。

打个比方：技能是提供给智能体的“专业能力说明书”（类似用户手册或操作指南）。执行任务时，智能体可按需加载相应技能，从而增强对任务的理解与执行能力。

### 2. Skill 的两大特点

- **结构化**：一个技能对应一个 `SKILL.md`，结构化描述任务目标、适用场景、关键约束、推荐步骤、可选脚本与模板。
- **动态按需加载**：智能体不会在任务开始时一次性读取所有技能全文。它会先扫描所有技能的简要描述，仅当判断当前任务与某技能高度相关时，才加载该技能的详细内容。**这种机制能有效减少 Token 消耗、避免无关信息干扰决策。**

> **🔍 Skill vs Rule 的关键区别：**
> 规则采用**全量加载**，一旦开启对话，所有规则都会被注入并持续占用上下文窗口；技能采用**按需加载**，仅在实际调用时才加载到上下文，显著降低 Token 消耗。
>
> 这就是为什么 superpowers 把 20 个 skill 做成 Skill 而不是 Rule——如果全塞进 Rule，上下文会被撑爆。

### 3. Skill 的结构

一个技能必须包含 `SKILL.md`，可选附加脚本、模板、参考文件：

```text
skill-name/
├── SKILL.md          # （必须）智能体的核心指令
├── examples/         # （可选）输入/输出示例
│   ├── input.md
│   └── output.md
├── templates/        # （可选）可复用模板
│   └── component.tsx
└── resources/        # （可选）参考文件、运行脚本
    └── style-guide.md
```

`SKILL.md` 的格式如下：

```markdown
---
name: 技能名称
description: 简要描述这个技能的功能和使用场景
---

# 技能名称

## 描述
描述这个技能的作用。

## 使用场景
描述触发这个技能的条件。

## 指令
清晰的分步说明，告诉智能体具体怎么做。

## 示例 (可选)
输入/输出示例，展示预期效果。
```

### 4. 全局技能 vs 项目技能

| 类型 | 目录 | 作用 |
|---|---|---|
| 项目技能 | 项目内 `.trae/skills/` | 注入项目专属业务知识、技术栈约束，随 Git 团队共享 |
| 全局技能 | `~/.trae/skills/`（macOS/Linux） | 跨项目复用，固化个人/组织长期偏好 |

在本课程中，我们用 `npx superpowers-zh` 把 skills 装到项目级 `.trae/skills/`，确保团队成员行为一致。

## 四、AGENTS.md：项目级行为指引

### 1. 它和 `.trae/rules/` 是什么关系

`AGENTS.md` 是项目根目录下的一个轻量级 Markdown 文件，用于向 AI 智能体提供行为指引。它通过直观、易读的文本描述，明确智能体在项目中需遵守的指令和规范。

**`AGENTS.md` 中定义的规则为项目级规则，仅在当前项目中生效。** 在 Trae 中创建的 `AGENTS.md` 会被自动纳入 `.trae/rules/` 体系，作为一条“始终生效”的项目规则参与 AI 行为管理。

> **为什么还要单独用 AGENTS.md？**
> 因为它是跨工具的事实标准——Claude Code、Cursor、Codex、Gemini CLI 等工具都认 `AGENTS.md`。用它写红线规矩，换工具也不用重写一遍。

### 2. 什么时候必须用 AGENTS.md

AI 不是项目负责人，你才是。如果不加约束地让 Trae 随意发挥，它可能擅自修改数据库表结构，或生成完全不符合 RESTful 规范的接口。因此，在让 Trae 介入开发之前，必须通过 `AGENTS.md` 给它立下严格的规矩。

`AGENTS.md` 适合放**项目红线**（必须遵守、不可违背的约束）：

- 文档至上：生成代码前必须先读 `docs/` 下的需求与设计文档
- 数据库红线：严禁擅自修改表结构
- 技术栈约束：后端 Spring Boot 3 + MyBatis-Plus，前端 Vue 3 + Element Plus
- 异常处理规范：业务异常统一抛 `BusinessException`
- 任务执行流程：先报思路、列改动文件，确认后再写代码

### 3. 标准的 AGENTS.md 模板

在项目根目录下创建（或完善）`AGENTS.md`，填入以下约束条款：

```markdown
# 项目开发 AI 行为规范

## 1. 核心原则
- **文档至上**：在生成任何代码前，必须优先阅读 `docs/` 目录下的相关文档（如 `requirement.md` 和 `design.md`）。
- **禁止擅自发挥**：绝不允许为了实现功能而擅自添加未在设计文档中出现的接口或功能。

## 2. 数据库约束
- 必须严格遵循 `docs/database.sql` 中的表结构和字段定义。
- 绝不允许擅自修改表结构（如增加字段、改变数据类型），如发现设计无法满足需求，必须提示开发者。

## 3. 技术栈与代码规范
- **后端**：使用 Spring Boot 3 + MyBatis-Plus。接口必须遵循 RESTful 风格，统一使用项目已有的 `Result<T>` 泛型类进行响应返回。
- **前端**：使用 Vue 3 (Composition API) + Element Plus。
- **异常处理**：所有业务异常必须抛出自定义的 `BusinessException`，由全局异常处理器接管。

## 4. 任务执行流程
当你接到新任务时，请严格按照以下步骤思考并行动：
1. 查找并阅读对应的需求文档和接口设计。
2. 给出你的开发思路和准备修改的文件列表。
3. 得到我（开发者）的确认后，再开始输出/修改代码。
```

### 4. 在 Trae 中应用这些规则

在 Trae 的对话（Chat）或 Builder/SOLO 模式中，下达第一个开发指令时，附带这样一句话：

> *"请先仔细阅读项目根目录的 AGENTS.md，以及 docs/ 目录下的 requirement.md，然后我们准备开始开发【图书借阅管理】模块。请先告诉我你的开发思路。"*

通过这种方式，你就为 Trae 戴上了“紧箍咒”，确保它生成的代码与系统设计保持高度一致。

## 五、Trae 中常用的 MCP 工具

前面讲了 Rules 和 Skills 管“规矩”，现在补上 MCP 管“能力”。MCP（Model Context Protocol）是给 AI 用的“万能工具箱”——让 AI 能操作浏览器、读取最新文档、管理代码仓库、跨会话记忆。

### 1. 必装·基础款（全场景通用）

| MCP Server | 作用 | 一句话推荐 |
|---|---|---|
| **Context7** | 实时抓取官方文档最新内容与指定版本示例 | 写新框架代码时必装，从根源避免用过时 API |
| **GitHub** | 操作仓库、Issue、PR、搜索代码 | 团队协作和版本管理基础 MCP |
| **Memory** | 跨会话持久记忆（知识图谱） | 长期项目必装，不用每次重复项目背景 |
| **File System** | 安全读写本地项目文件、配置、文档 | 全场景通用，省去复制粘贴 |

### 2. 前端开发·专属神器

| MCP Server | 作用 | 一句话推荐 |
|---|---|---|
| **Playwright** | 浏览器自动化：点击、截图、生成测试代码 | 前端验收、自动化测试必装 |
| **Chrome DevTools** | 开放 Chrome DevTools 全部能力 | 前端调试闭环最强 MCP 之一 |
| **Figma AI Bridge** | 读取设计稿布局、尺寸、颜色，导出图标 | UI 还原效率神器 |
| **Puppeteer** | 轻量浏览器自动化（导航、截图、执行 JS） | Playwright 的轻量替代，低配设备友好 |

### 3. 复杂推理·辅助款

| MCP Server | 作用 | 一句话推荐 |
|---|---|---|
| **Sequential Thinking** | 引导 AI 分步骤推理、拆解任务、记录中间判断 | 复杂问题分析必开 |
| **Fetch** | 拉取网页、接口或远程内容并结构化读取 | 轻量网页读取工具 |
| **Excel** | 读写 Excel 表格数据、生成公式 | 数据开发、用例管理场景 |

### 4. 安装与配额提醒

在 Trae 顶部标签栏点击 **MCP** 进入管理面板，可一键安装并启用/禁用各 MCP 工具。

> **⚠️ 工具上限 40 个：**
> 每启用一个 MCP 都会占用工具配额（系统上限 40 个工具）。全部启用很容易超限，应对策略：
> 1. 只启用核心 MCP（Context7、Fetch、Memory、Sequential Thinking 基本够用）；
> 2. 避免功能重叠（Playwright 和 Puppeteer 二选一）；
> 3. 不常用的直接关闭，按需开启；
> 4. 可在「智能体」标签页为每个智能体精细分配工具。

> **🔍 Skill vs MCP 的关键区别：**
> 技能描述 Trae **如何**完成一项任务，而 MCP Server 提供 Traa **可以调用**的工具。
> 例如：Playwright MCP 给 Trae 浏览器自动化**能力**；对应的 skill 则约定测试工程结构、POM 设计规范、测试编写流程，引导 Trae 在正确上下文中**高效调用**这些能力。

## 六、三层协同：本课程的最佳实践

回到本课程，我们这样组合使用：

1. **Rules 守底线** —— `AGENTS.md` 写入项目红线（文档至上、禁改表结构、技术栈约束），始终生效。
2. **Skills 给方法** —— `npx superpowers-zh` 安装 20 个 skill，覆盖 TDD、代码审查、系统化调试等工程方法论，按需触发。
3. **MCP 给能力** —— 按需启用 Context7（查最新文档）、Playwright/Chrome DevTools（前端验收测试）、Memory（跨会话记忆项目约定）。

三者分工后，AI 就从“自作主张、一路走到黑”的实习生，升级为“懂规矩、有方法、会动手”的资深工程师。
