# 课程项目选题库（topic-selection 资源）

> 本文件整理自课程电子教材的《项目选题库》，用于 `topic-selection` Skill 在帮助学生比较、推荐和缩小项目范围时参考。
>
> 使用规则：优先从本选题库中选择适合学生基础的题目；学生也可以提出自己的题目，但自拟题目的《项目选题立项书》必须经过任课教师审核后才能进入后续开发。

---

## 选题判断标准

选题不是“功能越多越好”，而是应当同时具备以下特点：

1. **有明确用户**：能说明系统具体给谁使用，而不是面向“所有人”；
2. **有真实问题**：能说明用户当前做法哪里不方便；
3. **有业务闭环**：至少有一条从开始到结束的完整流程；
4. **能按期完成**：核心版本适合课程周期，数据和技术条件基本可获得；
5. **能展示验证**：可以在答辩中演示、测试并解释。

### 难度说明

| 难度 | 适合情况 | 建议目标 |
| --- | --- | --- |
| **L1 基础** | 第一次独立完成 Web 项目，基础较弱 | 登录、基础管理、一条核心业务闭环 |
| **L2 标准** | 已做过简单增删改查，希望完成多角色业务 | 多角色权限、状态流转、统计或审核功能 |
| **L3 进阶** | 核心 Web 功能已经熟练 | 在完整业务系统上增加推荐、知识问答等 AI 能力 |

建议多数学生从 **L1 或 L2** 开始。先完成小而完整的系统，再增加特色功能。

## 范围控制原则

同一个题目，不同学生可以做出不同范围。建议将功能分为四类：

| 优先级 | 含义 | 示例 |
| --- | --- | --- |
| 必做 | 没有它就无法形成核心闭环 | 宿舍报修中的提交、派单、处理和确认 |
| 应做 | 能让系统更完整 | 查询筛选、状态记录、简单统计 |
| 选做 | 核心版本完成后再增加 | 智能推荐、消息提醒、数据可视化 |
| 本期不做 | 主动排除，避免项目失控 | 在线支付、复杂即时通信、跨校数据共享 |

基础项目建议控制在：

- **2～3 类用户**；
- **4～6 个核心模块**；
- **1～2 条主要业务流程**；
- **1 项以内的 AI 增强功能**。

如果一句话无法讲清主流程，通常说明范围还需要缩小。

## 自拟选题审核规则

学生可以提出选题库之外的项目方向，但应遵循以下规则：

1. 先使用 `topic-selection` Skill 检查目标用户、真实问题、核心闭环、数据与范围；
2. 完成《项目选题立项书》；
3. **自拟题目的《项目选题立项书》必须经任课教师审核通过后，才能进入原型、设计与开发阶段**；
4. 教师审核时重点检查：用户和问题是否真实、项目范围是否可控、AI 或外部服务是否有降级方案、成果是否可运行和可展示。

## 推荐策略

使用本 Skill 进行选题时：

- 如果学生没有明确想法，优先从本选题库中推荐 2～3 个适合其基础与场景的题目；
- 如果学生有候选题目，先与本选题库中类似项目比较，判断用户、闭环和范围；
- 如果学生题目明显过大，先缩小为本选题库类似的 L1/L2 版本；
- 如果学生坚持自拟题目，不直接否决，但必须提醒“立项书需任课教师审核”。
