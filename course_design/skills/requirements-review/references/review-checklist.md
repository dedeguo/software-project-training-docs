# 两份项目文档评审清单

仅在需要完整评审《项目选题立项书》和《需求分析说明书》时读取。按项目实际情况检查，不相关项可标记“不适用”，不得为了填满清单而新增需求。

## 一、立项书检查

| 检查项 | 通过标准 |
| --- | --- |
| 项目名称与介绍 | 一句话能说明目标用户、问题和项目作用 |
| 背景与问题 | 描述用户当前做法及具体困难，不只罗列技术 |
| 目标用户与场景 | 用户具体，场景能说明何时使用以及期望结果 |
| 调研依据 | 用户交流或同类产品体验真实、可说明来源，不编造数量或结论 |
| 项目目标 | 与用户问题对应，不写无法判断的空话 |
| 核心业务闭环 | 有发起、处理和用户获得结果 |
| 必做范围 | 每项都直接支撑核心闭环 |
| 选做与本期不做 | 清楚排除非核心、高风险或条件不足内容 |
| 实施计划 | 每阶段都有可检查的文档、功能、测试或部署成果 |
| 风险与应对 | 风险具体，应对办法可执行 |
| AI 使用 | AI 不作为虚构调研来源，生成内容已经人工确认 |

## 二、需求说明书检查

| 检查项 | 通过标准 |
| --- | --- |
| 项目概述 | 背景、目标和范围与立项书一致 |
| 用户角色 | 通常 1～3 类，目标、主要操作和不能做的操作明确 |
| 核心场景 | 通常 3～5 个，包含触发、目标、过程、结果和异常 |
| 功能需求 | 必做功能有编号、角色、前置条件、输入、处理、结果和异常 |
| 业务流程 | 有开始、判断、失败分支、状态变化和结束 |
| 状态说明 | 明确当前状态、允许操作者、允许操作和下一状态 |
| 业务规则 | 权限、输入、数据、状态和重复操作等规则具体可判断 |
| 验收条件 | 每个必做功能至少有正常及异常/边界场景，结果可观察 |
| 本期不做 | 与立项书一致，未在其他章节重新出现为必做 |
| 变更记录 | 如范围发生变化，说明原因、影响和确认情况 |

## 三、跨文档一致性检查

| 对照项 | 需要发现的问题 |
| --- | --- |
| 项目名称 | 名称、版本或项目对象不一致 |
| 用户 | 需求中出现立项书未说明的新角色，或遗漏核心用户 |
| 问题与目标 | 功能无法解决立项书中的问题，目标被悄然改变 |
| 核心闭环 | 发起角色、处理步骤、最终结果不一致 |
| 必做功能 | 需求新增立项无依据的必做功能 |
| 优先级 | 选做或本期不做被改成必做 |
| 排除项 | 被排除内容又出现在流程、规则或验收中 |
| 外部依赖 | 立项承诺降级，需求却强制依赖 AI 或第三方服务 |
| 周期与计划 | 承诺范围不能由计划中的阶段成果覆盖 |

## 四、闭环与追踪检查

对每条主流程进行以下追踪：

```text
立项目标 → 目标用户 → 核心场景 → 必做功能
→ 流程步骤与状态 → 业务规则 → 正常验收 + 异常/边界验收
```

任一环缺失时，指出准确断点。不要要求学生额外提交复杂追踪表。

## 五、常见严重问题

- 用户能够查看或修改他人数据，但文档没有授权规则；
- 普通用户与管理员的权限描述互相冲突；
- 流程中出现审核、派单、确认等步骤，功能表没有对应功能；
- 已完成或已取消记录仍可重复处理；
- 状态名称在流程图、规则和验收中不一致；
- 一个必做功能只有“支持管理”等菜单描述；
- 验收条件只有主观词，没有可观察结果；
- 立项书列为选做或不做的内容，在需求中变成必做；
- AI、支付、短信、地图或硬件不可用时，核心业务无法运行；
- 调研结论没有来源，或明显由 AI 推测生成。

## 六、快速提交门槛

- 没有未解决的范围冲突、闭环断裂、权限或状态矛盾；
- 所有材料事实和调研证据由学生确认；
- 每个必做功能能追踪到立项目标、业务规则和验收条件；
- 必做功能至少有正常和异常/边界验收场景；
- 所有疑似超范围内容已降级、排除或经教师确认；
- 学生能够在 3 分钟内说明用户、问题、主流程、必做范围和完成标准。
