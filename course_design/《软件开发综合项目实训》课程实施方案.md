# 《软件开发综合项目实训》课程实施方案

> **实施总思路：**统一开发过程、差异化项目选题、分层任务要求、AI辅助开发、阶段成果验收。
> 

课程不采用“教师带领全班复制同一个项目”的传统方式，也不让学生在缺少边界的情况下完全自由开发。教师统一控制开发流程、课堂节奏、项目规范和验收标准，学生根据兴趣和已有基础选择不同项目，在AI辅助下逐步完成自己的软件作品。

## 一、实施目标

课程实施应围绕四个直接目标展开：

| 目标 | 实施重点 |
| --- | --- |
| 学生有兴趣 | 允许学生在教师审核范围内选择与校园、生活、行业或个人发展相关的真实项目 |
| 学生听得懂 | 先展示成果、再拆解任务，只讲当前项目阶段真正需要的知识 |
| 学生做得出来 | 提供脚手架、任务单、示例、AI助教和阶段反馈，降低项目启动与调试门槛 |
| 学生有成就感 | 每周形成可见成果，最终完成部署、路演和优秀作品展示 |

核心学习链路是：

> 选择一个愿意做的项目 → 每节课完成一个小任务 → 每周看到项目发生变化 → 最终交付一件完整作品。
> 

## 二、开课前的准备工作

教师不必在开课前把全部课程资源一次性制作完成，但至少应准备以下五类基础材料。

### 1. 一个完整的参考项目

参考项目用于展示软件开发全过程和课堂示范，不要求学生原样复制。项目应具有：

- 明确的目标用户和使用场景；
- 登录、权限和核心业务流程；
- 完整的数据库与前后端交互；
- 基本测试、部署和项目文档；
- 一个难度适中的AI或特色功能。

### 2. 一套可运行的项目脚手架

脚手架可包含：

- 基础项目结构和依赖配置；
- 数据库连接与基础数据；
- 统一返回、异常处理和参数校验示例；
- 登录、文件上传等常用功能；
- 前端基础页面与接口调用示例；
- Git、接口测试和部署说明。

脚手架的作用是帮助学生跨过环境配置和项目启动等高频障碍，而不是替学生完成核心业务。

### 3. 一个分级项目选题库

项目方向可以包括：

- 校园服务类：失物招领、宿舍报修、社团活动、校园二手交易；
- 生活服务类：健康管理、宠物救助、个人记账、志愿服务；
- 行业应用类：旅游推荐、社区服务、预约管理、商品管理；
- AI应用类：AI学习助手、AI简历优化、课程知识问答、智能推荐。

项目选题需要审核其真实性、业务闭环、工作量、数据来源和16周内的可完成性。

### 4. 一套过程文档模板

包括：

- 项目立项书；
- 需求分析文档；
- 原型与系统设计说明；
- 数据库和接口设计模板；
- 测试报告与缺陷记录；
- 项目周报；
- AI协作记录；
- README、答辩PPT和个人总结模板。

### 5. 一套阶段验收标准

每个阶段都应明确：

- 学生需要提交什么；
- 课堂上需要演示什么；
- 哪些功能必须运行；
- 学生需要解释哪些内容；
- 达到什么条件才能进入下一阶段。

## 三、项目组织方式

### 1. 个人项目与组队要求

原则上鼓励学生个人独立完成项目。只有当项目功能模块、角色数量和业务复杂度达到一定规模时，才允许组队。

组队要求：

- 每组不超过3人；
- 项目内容和工作量须经教师审核；
- 必须明确成员分工和接口边界；
- 使用Git提交记录个人真实贡献；
- 最终答辩需要分别进行个人提问和代码解释。

### 2. 技术路线

推荐使用Spring Boot + Vue，但不将技术框架作为统一硬性要求。

学生可以根据项目需要和已有基础，选择Spring Boot、Servlet、JDBC、HTML、CSS、JavaScript等技术。课程统一的是开发过程、工程规范和交付标准，而不是所有学生必须使用完全相同的技术。

### 3. 项目范围控制

选题审核重点检查：

1. 目标用户是否明确；
2. 要解决的问题是否真实；
3. 是否具有完整业务流程；
4. 工作量是否与个人或团队规模匹配；
5. 数据来源是否合法；
6. 是否能够在16周内完成。

项目应优先完成最小可运行版本和核心业务闭环，再增加统计分析、AI问答、智能推荐等扩展功能。

## 四、16周课程推进安排

| 阶段 | 周次 | 核心任务 | 阶段成果 |
| --- | --- | --- | --- |
| 项目启动 | 第1—2周 | 课程导入、优秀作品展示、AI编程体验、项目调研与选题审核 | 项目立项书 |
| 需求与设计 | 第3—4周 | 用户分析、功能清单、业务流程、页面原型、数据库与权限设计 | 需求与系统设计文档 |
| 项目搭建 | 第5—6周 | 开发环境、Git仓库、数据库、项目骨架、登录或基础访问功能 | 可运行项目骨架 |
| 核心开发 | 第7—11周 | 核心模块、角色权限、业务状态流转和完整业务流程 | 可演示的业务闭环 |
| 完善交付 | 第12—15周 | 特色功能、测试修复、性能与交互优化、部署上线和成果包装 | 可访问的完整项目 |
| 展示评价 | 第16周 | 项目路演、个人答辩、代码解释和课程复盘 | 最终作品与个人总结 |

## 五、各阶段具体执行

### 第一阶段：项目启动

第一周不急于讲授大量技术知识，而是让学生首先看到课程最终能够做成什么。

课堂安排：

- 展示2—3个完整项目；
- 演示系统的登录、业务流程、部署地址和项目文档；
- 展示普通项目如何增加AI问答、推荐或数据分析功能；
- 说明项目与毕业设计、学科竞赛和求职经历的关系；
- 体验一次“描述需求—AI提出方案—生成代码—运行验证—修改完善”的过程；
- 发布项目选题库、课程规则和最终成果清单。

第2周完成选题审核。学生提交项目名称、目标用户、核心问题、主要功能、技术方案以及组队分工。

### 第二阶段：需求与系统设计

这一阶段不要求撰写冗长文档，重点是帮助学生把项目想清楚。

最低成果包括：

- 目标用户和使用场景；
- 核心问题与项目目标；
- 功能清单；
- 一个完整业务流程；
- 页面原型；
- 数据库表；
- 用户角色和权限。

教师应在这一阶段重点控制项目范围，避免学生设计大量功能却无法完成核心流程。

### 第三阶段：项目搭建

学生需要完成：

- 创建Git仓库；
- 搭建项目和配置开发环境；
- 连接数据库；
- 让项目成功启动；
- 完成登录或基础访问功能；
- 形成第一次有效Git提交。

这一阶段的核心验收标准是：**项目必须真正运行起来。**

### 第四阶段：核心业务开发

教师将项目开发拆分成连续的小任务：

> 第一个增删改查模块 → 用户角色与权限 → 核心业务流程 → 状态流转 → 查询统计。
> 

例如，宿舍报修系统的业务闭环可以是：

> 学生提交报修 → 管理员分配维修人员 → 维修人员处理 → 学生确认与评价。
> 

只有完成这种跨角色的完整流程，才算形成业务闭环。若项目只有若干互不关联的管理页面，不能认定为完整项目。

第8周设置中期演示，重点检查：

- 项目是否能够运行；
- 是否完成第一个核心模块；
- 项目范围是否仍然合理；
- 团队分工是否真实；
- 后续任务是否清晰。

### 第五阶段：完善与交付

核心业务完成后，再进行：

- AI问答、智能推荐、统计分析等特色功能；
- 参数校验、异常处理和权限完善；
- 功能、异常和边界测试；
- 缺陷修复和界面优化；
- 项目部署上线；
- README、截图、视频、PPT和简历描述制作。

AI功能属于进阶或特色任务，不能因为追求AI创新而导致基础业务未完成。

### 第六阶段：项目答辩

答辩应同时考察成果质量和真实掌握情况：

1. 现场运行项目；
2. 演示完整业务流程；
3. 随机指定一个功能进行讲解；
4. 解释数据库设计和关键代码；
5. 说明AI参与了哪些工作；
6. 展示遇到的问题及解决过程；
7. 组队项目分别说明个人贡献。

必要时可以要求学生现场完成一个小修改，例如增加字段、调整校验规则或修改查询条件，以检验其真实理解和修改能力。

## 六、每次4课时的课堂组织

| 课堂环节 | 建议时间 | 主要内容 |
| --- | --- | --- |
| 成果展示与任务说明 | 15分钟 | 展示本周任务完成后的效果，明确验收标准 |
| 关键知识讲解 | 30分钟 | 讲解完成当前任务所需的方法、规范和常见问题 |
| 教师现场演示 | 30分钟 | 完成一个代表性功能，并演示错误定位过程 |
| 学生项目实践 | 75分钟 | 把课堂方法应用到自己的项目中 |
| 辅导、展示与验收 | 30分钟 | 个别辅导、运行检查、学生展示和Git提交 |

课堂实施的基本要求是：

> 教师讲完以后，学生立即动手；学生做完以后，立即运行验证；形成结果以后，及时反馈和验收。
> 

## 七、每周成果与过程记录

每周只设置一个清晰、可完成、可验证的主要成果，例如：

- 完成项目立项书；
- 完成页面原型；
- 完成数据库设计；
- 让项目成功运行；
- 完成登录功能；
- 完成第一个业务模块；
- 跑通一个核心业务流程；
- 完成测试并修复问题；
- 完成部署上线。

学生每周提交：

- 当前运行截图或演示结果；
- Git提交记录；
- 本周完成内容；
- 遇到的主要问题及解决情况；
- 下一周计划。

周报应简洁，重点保留真实过程证据，避免过程材料占用过多开发时间。

## 八、分层任务设计

| 层次 | 任务要求 |
| --- | --- |
| 保底任务 | 跟随模板完成基础功能，保证项目能够运行 |
| 标准任务 | 独立完成本项目对应业务功能，形成完整业务闭环 |
| 挑战任务 | 增加AI、推荐、数据分析、可视化或工程化能力 |

基础较弱的学生能够完成一个小而完整的项目；大多数学生能够完成可部署的业务系统；能力较强的学生可以加入AI功能，并继续发展为毕业设计、竞赛作品或求职项目。

## 九、AI辅助开发的执行规范

允许学生使用Trae、Codex、Cursor等AI编程工具，但必须遵循：

> 明确任务 → 补充上下文 → AI提出方案 → 人工审核 → 分步实现 → 运行测试 → 修正问题 → Git提交。
> 

学生不需要记录所有AI对话，但应记录关键协作节点：

- AI协助完成了什么；
- 提供了哪些项目上下文和约束；
- AI生成内容是否可以直接运行；
- 出现了什么错误；
- 学生进行了哪些判断和修改；
- 最终如何验证结果。

AI生成内容的有效性采用三个判断标准：

- **能否运行；**
- **能否解释；**
- **能否修改。**

不能运行、不能解释或没有经过验证的AI生成内容，不认定为学生的有效项目成果。

## 十、教师角色转变

教师不再是全程编写代码、让学生照着输入的讲授者，而是承担以下角色：

- **项目经理：**控制项目范围和总体进度；
- **架构顾问：**审核技术方案和业务设计；
- **开发教练：**示范任务拆分、调试和问题解决；
- **质量负责人：**组织阶段验收和质量检查；
- **作品策展人：**遴选、展示和推广优秀项目。

课堂上最重要的工作，是发现学生卡在哪里，并将问题转化为下一步可以执行和验证的小任务。

## 十一、考核评价建议

| 考核内容 | 建议比例 | 评价重点 |
| --- | --- | --- |
| 阶段任务与过程验收 | 30% | 每周成果、进度真实性和持续改进 |
| 需求、设计和测试文档 | 15% | 文档是否与实际项目一致 |
| 项目功能与业务闭环 | 20% | 系统是否运行，业务流程是否完整 |
| AI协作与问题解决 | 15% | 能否审查、修改、调试和验证AI输出 |
| 部署、展示与答辩 | 15% | 能否完成交付、演示和解释项目 |
| 团队协作与个人贡献 | 5% | 任务分工、Git记录和个人真实贡献 |

过程成绩应高于最终一次答辩，避免学生在课程末期依赖AI临时生成项目，也鼓励学生持续推进。

## 十二、风险控制与止损机制

### 1. 项目范围过大

优先保留核心用户、核心场景和核心流程，暂缓非必要模块和高级功能。

### 2. 学生连续两周没有进展

启用教师脚手架、缩减功能范围，将任务重新拆分为可在一周内完成的小目标。

### 3. 过度依赖AI

通过随机代码讲解、现场小修改、运行测试和Git记录检查学生真实掌握情况。

### 4. 团队成员贡献不均

结合成员分工、Git提交、项目周报、中期汇报和个人答辩分别评价。

### 5. 追求AI功能导致基础项目未完成

AI功能必须安排在核心业务闭环完成之后。基础项目未完成时，暂停AI扩展任务。

> **止损原则：**宁可完成一个小而完整、能够运行和讲清楚的项目，也不要留下一个功能很多但无法交付的半成品。
> 

## 十三、首次实施建议

第一轮课程不必追求项目类型过多或技术体系过于复杂，可以采用最小可行方案：

1. 准备1个教师参考项目；
2. 提供2—3套基础脚手架；
3. 建立10—15个经过审核的项目选题；
4. 设置5个关键验收节点：立项、设计、项目启动、中期演示、最终答辩；
5. AI功能作为进阶任务，不要求所有学生必须完成；
6. 课程结束后根据学生卡点、完成率和作品质量迭代教材与模板。

## 十四、关键执行原则

1. 选题必须审核，防止范围失控；
2. 每周必须产生一个可以运行或验证的成果；
3. 第8周必须进行中期演示，及时调整或缩减项目；
4. 先完成核心业务闭环，再增加AI或特色功能；
5. 技术路线可以多样，但过程规范和交付标准必须统一；
6. 最终必须部署、演示和解释，不能只提交代码；
7. 最终交付的不是一次性作业，而是一件能够展示和持续迭代的软件作品。
