# 《软件开发综合项目实训》教学项目案例体系与使用方案
> **总体思路：**课程不应让一个项目承担所有教学任务，而应建立“入门体验—教师示范—学生实践—AI进阶”的分层项目体系。
> 

通过不同项目承担不同教学功能，使学生先通过小项目建立信心，再通过教师参考项目学习规范开发方法，随后迁移到自主项目形成综合能力，最后通过AI进阶案例打开技术视野。

## 一、教学项目案例体系

| 项目层次 | 推荐项目 | 主要作用 | 使用阶段 |
| --- | --- | --- | --- |
| AI编程体验项目 | 校园图书管理系统 | 快速体验AI编程，建立信心 | 第1周 |
| 教师贯穿参考项目 | 校园集市——校园二手物品交易与智能推荐平台 | 示范完整软件开发过程和通用开发方法 | 第2—15周 |
| 学生自主项目 | 校园服务、生活服务、行业应用或AI应用项目 | 学生迁移实践，形成课程主要考核作品 | 第2—16周 |
| AI进阶案例 | CourseMind AI课程智能体平台 | 展示RAG、学习画像、多智能体和AI产品能力 | 第12周及课外拓展 |

项目之间的递进关系是：

> 小项目建立信心 → 参考项目学习方法 → 自主项目形成能力 → AI项目打开上限。
> 

## 二、项目一：校园图书管理系统

### 1. 项目定位

校园图书管理系统用于课程开始阶段的AI编程体验，不作为16周贯穿项目，也不要求学生完成完整的软件开发全过程。

本项目直接向学生提供需求、数据库和基本设计，使学生暂时跳过需求分析与系统设计，集中体验AI辅助编码、调试、测试和Git提交。

项目仓库：校园图书管理系统

### 2. 第一周的教学目标

学生通过一个较小任务认识AI编程的基本过程：

1. 阅读教师提供的需求和设计文档；
2. 将功能拆分为可执行的开发任务；
3. 向AI补充技术环境、已有代码和验收要求；
4. 让AI提出实现计划；
5. 分步骤生成或修改代码；
6. 编译、运行并处理错误；
7. 完成功能测试；
8. 使用Git提交；
9. 演示成果并说明AI参与了什么。

### 3. 功能范围

第一周不要求完成整个图书管理系统，可以从以下任务中选择一个：

- 图书列表；
- 新增图书；
- 按书名查询；
- 用户登录；
- 图书借阅；
- 图书归还。

### 4. 预期效果

这一阶段最重要的不是代码数量，而是让学生形成第一次正向反馈：

> 我能够借助AI，在一节课内完成一个真正可以运行的功能。
> 

## 三、项目二：校园集市教师参考项目

### 1. 项目来源与定位

教师贯穿参考项目建议以现有电商系统为基础进行教学化精简，不必从零重新开发。

现有项目：生活集·电商平台

参考项目可改造为：

> **校园集市——校园二手物品交易与智能推荐平台**
> 

它保留电商系统典型的用户、分类、信息发布、图片上传、查询、交易、状态流转和后台管理能力，但删除商业电商中对学生负担较大的SKU、多支付渠道和复杂退款等内容。

### 2. 为什么选择校园集市

- 业务与学生校园生活密切相关，容易理解；
- 页面效果直观，容易形成学习兴趣；
- 具有多角色和完整业务流程；
- 能覆盖软件综合实训的大部分技术与工程能力；
- 可以由基础业务自然扩展到AI生成、智能分类和推荐；
- 现有电商系统已经具备代码、设计文档、接口文档、测试和AI开发记录，改造成本较低。

### 3. 核心角色

- **普通学生：**注册登录、发布闲置物品、浏览查询、发起交易、确认完成和评价；
- **管理员：**管理分类、审核内容、处理违规信息和查看统计数据。

普通学生可以同时承担买方和卖方角色，不再单独设计复杂的SELLER角色。

### 4. 核心业务闭环

> 学生注册登录 → 发布闲置物品 → 管理员审核 → 其他学生浏览查询 → 提交购买或预约 → 发布者确认 → 线下交易 → 双方确认完成 → 评价。
> 

这个业务闭环能够覆盖：

- 用户与权限；
- 基本增删改查；
- 图片上传；
- 分类、搜索和分页；
- 多角色协作；
- 业务状态流转；
- 异常处理；
- 数据统计；
- 测试；
- 部署；
- AI特色扩展。

### 5. 教学版保留内容

- 注册和登录；
- JWT或Session认证；
- 用户信息；
- 商品或闲置物品分类；
- 信息发布与图片上传；
- 列表、详情、搜索和分页；
- 收藏或交易清单；
- 交易申请；
- 交易状态流转；
- 管理员审核；
- 统一返回与异常处理；
- Git、接口测试和部署。

### 6. 教学版精简内容

第一版建议删除或弱化：

- SPU和SKU多规格；
- 笛卡尔积生成SKU；
- 复杂库存并发；
- 独立卖家角色；
- 支付宝真实支付回调；
- 退款和售后流程；
- 过多设计模式；
- 过高的测试覆盖率门槛。

支付可以简化为“模拟支付成功”或“线下交易＋双方确认”，保留订单状态流转即可。

### 7. 分阶段版本设计

| 版本 | 主要内容 | 教学作用 |
| --- | --- | --- |
| V0 可运行骨架 | 前后端项目、数据库连接、统一返回、全局异常和一个基础接口 | 让所有学生先把项目运行起来 |
| V1 基础管理 | 注册登录、权限、分类、物品发布、图片上传、列表和详情 | 学习认证、数据库和基础增删改查 |
| V2 业务闭环 | 收藏或交易清单、交易申请、确认、状态流转、完成、评价和审核 | 学习多角色业务流程 |
| V3 质量改进 | 参数校验、异常处理、接口测试、缺陷修复、界面优化和部署 | 培养质量与项目交付能力 |
| V4 AI特色 | 描述生成、智能分类、自然语言搜索、推荐或规则问答 | 学习AI能力与业务系统集成 |

AI功能不要求全部实现，应选择一个稳定、易理解、便于演示的功能。

## 四、项目三：学生自主项目

### 1. 项目定位

学生自主项目是课程的主线作品和主要考核对象。教师参考项目只负责示范方法，学生必须把方法迁移到自己的业务场景中。

推荐方向：

- **校园服务类：**宿舍报修、失物招领、社团活动、校园预约；
- **生活服务类：**健康管理、宠物救助、个人记账、志愿服务；
- **行业应用类：**旅游推荐、社区服务、预约管理、资源管理；
- **AI应用类：**AI学习助手、AI简历优化、知识库问答、智能推荐。

### 2. 参考项目与学生项目的迁移关系

| 教师参考项目 | 学生项目中的对应功能 |
| --- | --- |
| 闲置物品发布 | 报修提交、活动发布、救助信息发布、预约申请 |
| 商品分类 | 报修分类、活动分类、资源分类、服务分类 |
| 收藏或交易清单 | 收藏夹、申请清单、预约清单、任务清单 |
| 交易订单 | 报修工单、预约单、申请单、救助任务 |
| 交易状态 | 处理状态、审核状态、任务状态、服务状态 |
| 交易评价 | 服务评价、活动反馈、维修评价、救助反馈 |
| 管理员审核 | 内容审核、申请审核、项目审核、信息审核 |

教师演示的是信息发布、权限控制、状态流转和异常处理等通用方法，而不是要求所有学生都实现商品和订单。

### 3. 组织要求

- 原则上鼓励个人完成；
- 功能和业务复杂度达到要求后才允许组队；
- 每组不超过3人；
- 项目内容、工作量和成员分工须经教师审核；
- 组队项目必须通过Git提交、项目周报和个人答辩说明真实贡献。

## 五、项目四：CourseMind AI进阶案例

### 1. 项目定位

CourseMind AI不适合作为全班从头跟做的主参考项目，因为其涉及Next.js、Spring AI、Redis、Milvus、MinIO、RAG、学习画像和多智能体等内容，整体复杂度较高。

项目仓库：CourseMind AI课程智能体平台

它更适合承担以下作用。

### 2. 优秀项目展示

第一周可以展示CourseMind AI，让学生看到普通管理系统之外的AI软件形态，理解课程项目未来可以向毕业设计、竞赛或产品方向发展。

### 3. AI功能案例

第12周可以选择其中相对独立的内容进行演示：

- 大模型API调用；
- 流式问答；
- RAG知识库；
- Prompt管理；
- AI服务模块封装。

不需要在基础课程中完整讲授学习画像、多智能体协商和复杂基础设施。

### 4. 进阶学生参考

能力较强的学生可以参照CourseMind AI完成：

- AI课程学习助手；
- 课程知识库问答；
- AI毕业设计助手；
- 个性化学习资源生成；
- AI项目导师。

### 5. 课程AI助教平台

CourseMind AI还可以直接服务本课程：

- 回答电子教材问题；
- 根据学生项目生成阶段任务建议；
- 分析需求和数据库设计；
- 诊断代码错误；
- 提供测试建议；
- 审查项目进度；
- 模拟项目答辩。

## 六、参考项目在课程中的使用方法

参考项目采用“双线并行”：

> **教师参考项目负责示范方法，学生自主项目负责迁移实践。**
> 

每周的基本流程是：

1. 教师展示参考项目本周完成后的效果；
2. 说明对应的需求、设计和验收标准；
3. 使用AI完成一个代表性开发任务；
4. 学生在自己的项目中实现相同类型的业务能力；
5. 课堂结束前进行运行检查、代码解释和Git提交。

## 七、一次课堂示例

假设本周任务是“信息发布功能”。

### 教师在校园集市中演示

1. 编写“发布闲置物品”的用户故事；
2. 确定标题、分类、价格、图片和描述等字段；
3. 设计数据库表；
4. 设计新增接口；
5. 向AI提供项目背景、已有代码和验收标准；
6. 让AI提出实现计划；
7. 分步骤生成或修改代码；
8. 运行并使用Apifox测试；
9. 处理异常和参数错误；
10. 使用Git提交。

### 学生迁移

- 报修项目：实现“提交报修”；
- 社团项目：实现“发布活动”；
- 健康项目：实现“新增健康记录”；
- 宠物救助项目：实现“发布救助信息”；
- 预约项目：实现“提交预约申请”。

### 当堂验收

- 页面或接口是否能够运行；
- 数据是否正确写入数据库；
- 参数错误时是否得到合理处理；
- 学生能否解释主要实现；
- 是否形成有效Git提交。

## 九、参考项目仓库组织

教师参考项目不应只提供一个最终版本，而应保留不同阶段的可运行版本。

建议采用：

- `starter`分支：可运行的基础脚手架；
- `main`分支：最终完整参考项目；
- `v1-auth`标签：登录认证；
- `v2-basic-crud`标签：基础业务；
- `v3-workflow`标签：完整业务闭环；
- `v4-quality`标签：测试与质量改进；
- `v5-ai`标签：AI特色功能；
- `v6-release`标签：部署交付版。

每个阶段同时提供：

- 阶段需求；
- 验收标准；
- AI提示词参考；
- 关键代码说明；
- 常见错误；
- 测试方法；
- 成果截图。

## 十、如何避免照抄参考项目

课程不以“完全不参考代码”为目标，而是要求学生能够理解、迁移和修改。

可以采用以下措施：

- 学生项目业务主题与教师参考项目不同；
- 项目选题、功能和数据库必须经过审核；
- 每周检查运行结果与Git提交；
- 随机指定代码进行解释；
- 要求说明AI生成内容的修改过程；
- 第8周进行中期业务闭环检查；
- 最终答辩安排现场小修改；
- 组队项目分别进行个人答辩。

学生成果采用三个基本判断标准：

> **能运行、能解释、能修改。**
> 

参考代码后能够理解并迁移到自己的业务中，属于有效学习；直接复制且无法解释，则不能认定为有效成果。

## 十一、最终实施建议

课程项目体系建议确定为：

1. **校园图书管理系统：**第一周AI编程快速体验；
2. **校园集市教学版：**教师贯穿16周的参考示范项目；
3. **学生自主项目：**课程主要实践和考核成果；
4. **CourseMind AI：**AI进阶案例、优秀作品展示和课程智能助教平台。

这样既能降低学生起步难度，又能避免所有学生完成相同项目，同时保留软件工程、AI协作、毕业设计、竞赛和求职成果之间的衔接。

## 六、课程实施总体思路：双项目、文档驱动、AI辅助

本课程采用“**教师带练项目＋学生自主项目**”的双项目教学模式，以规范的软件开发过程为主线，以AI编程工具为辅助，通过“教师示范—学生模仿—成果提交—阶段评审—修改完善”的方式推进。

> **课程核心理念：**不是让AI替学生随意生成一个系统，而是让学生先通过需求和设计明确“要做什么、准备怎么做”，再让AI严格依据文档完成开发。文档既是团队协作的依据，也是约束AI、检查代码和开展测试的重要标准。
> 

课程整体流程为：

> 图书管理系统带练 → 学生项目选题 → 项目立项 → 需求分析 → 系统设计 → 工程搭建 → AI辅助迭代开发 → 系统测试 → 部署上线 → 项目答辩
> 

### 1. 教学模式

- **第一个项目：校园图书管理系统。**由教师提供较完整的需求、设计、数据库、接口和测试部署资料，带领学生快速体验一次完整的软件开发过程。
- **第二个项目：学生自主项目。**学生在教师提供的范围内选题，也可以自主提出题目，但必须经过教师审核。学生将图书管理系统中学到的方法迁移到自己的项目中。
- **教学实施方式：**每个阶段均先由教师结合案例示范，再由学生完成自己的阶段任务，提交成果并接受审核。
- **课程评价重点：**不仅看最终功能，还要看需求、设计、代码、测试和部署是否一致，是否具有真实的开发过程，以及学生能否解释AI生成和修改的代码。

## 七、第一阶段：校园图书管理系统AI开发体验

### 1. 阶段定位

图书管理系统是课程开始阶段的带练项目。该阶段不要求学生从零完成需求分析和系统设计，而是由教师直接提供相关文档，让学生集中体验：

- 阅读和理解需求、设计文档；
- 根据文档拆分开发任务；
- 使用AI搭建和修改项目；
- 检查AI生成代码；
- 完成功能测试和问题修复；
- 将项目部署运行；
- 使用Git记录开发过程。

这一阶段的目标是让学生建立第一次完整的软件项目经验和AI编程信心。

### 2. 教师提供的资料

教师可向学生提供：

- 《图书管理系统需求分析说明书》；
- 《图书管理系统设计说明书》；
- UI页面原型或页面示意图；
- 数据库设计和初始化脚本；
- 接口设计说明；
- 项目脚手架；
- 编码规范和项目规则文件；
- 测试要求或部分测试用例；
- 部署说明模板。

### 3. 教学过程

#### （1）先展示完整系统

教师首先演示图书管理系统最终效果，包括登录、图书管理、借阅归还、用户管理和数据统计等功能，让学生明确最终目标。

#### （2）带领学生阅读文档

教师结合具体功能说明需求、设计和代码之间的关系。例如：

> 需求文档规定“只有管理员可以删除图书”；设计文档进一步规定删除接口、权限校验和异常提示；最终代码必须按照这些要求实现，测试阶段也应验证普通用户不能删除图书。
> 

#### （3）让AI遵循项目文档

项目中建议设置统一的文档目录和AI规则文件，例如：

```
项目根目录
├── docs
│   ├── requirement.md
│   ├── design.md
│   ├── database.md
│   ├── api.md
│   └── ui.md
├── AGENTS.md
└── README.md
```

在规则文件中明确：

- 使用的技术和版本；
- 工程目录结构；
- 数据库表和字段；
- 接口格式和命名规范；
- UI和交互要求；
- 不允许随意增加功能；
- 不允许擅自修改数据库和接口；
- 开发前必须先阅读相关文档。

#### （4）分模块完成开发

不要求学生直接让AI生成整个系统，而是将项目拆分为工程初始化、数据库配置、登录认证、图书管理、借阅管理、后台管理和前后端联调等任务，逐步完成、运行和提交。

#### （5）完成测试和部署

教师演示如何根据需求编写测试用例，测试正常流程、异常输入和权限控制，记录Bug，使用AI辅助定位问题，并在修复后进行回归测试。最后完成数据库初始化、项目打包和部署运行。

### 4. 第一阶段学生成果

学生提交：

- 图书管理系统源代码；
- Git提交记录；
- 关键功能运行截图；
- 简要测试记录；
- 部署运行结果；
- AI辅助开发体验总结。

体验总结重点说明：AI完成了哪些工作、哪些代码经过人工修改、文档不清楚时AI出现了什么问题，以及如何通过文档和提示使AI更准确地完成任务。

## 八、第二阶段：学生项目选题与立项

### 1. 选题方式

学生可以从校园服务、生活服务、行业应用和AI应用等范围中选择项目，也可以自主提出题目。所有题目必须经过教师审核后才能立项。

### 2. 审核重点

教师重点审核：

- 是否具有明确用户和真实应用场景；
- 是否具有完整的业务流程；
- 功能量是否与个人或团队人数匹配；
- 是否能够在课程周期内完成；
- 是否只是简单页面和数据表的堆积；
- 是否存在明显超出学生能力的复杂技术；
- 是否与其他小组高度重复。

原则上鼓励个人完成；确有足够业务复杂度时可以组队，每组不超过3人。

### 3. 阶段成果

学生提交《项目申报与计划书》，主要包括：

- 项目背景和意义；
- 目标用户和应用场景；
- 核心功能；
- 初步技术路线；
- 团队成员分工；
- 项目实施计划；
- 预期成果。

项目立项审核通过后进入需求分析阶段。

## 九、第三阶段：需求分析

### 1. 教师示范内容

教师以图书管理系统或校园集市项目为例，示范：

- 如何识别用户角色；
- 如何分析用户需求和业务场景；
- 如何划分功能模块；
- 如何描述正常流程和异常流程；
- 如何编写功能需求和非功能需求；
- 如何绘制用例图和业务流程图；
- 如何判断需求是否明确、完整和可测试。

需求描述不能只是功能名称的简单罗列，而应说明“谁在什么情况下进行什么操作，系统产生什么结果”。

### 2. AI在需求阶段的作用

AI可以辅助整理表达、发现遗漏、补充异常流程和检查需求冲突，但不能凭空决定系统业务。学生必须先明确项目背景、用户角色、业务场景、核心功能和约束条件，再使用AI完善文档。

### 3. 阶段成果

学生提交：

- 《软件需求分析说明书》；
- 用例图；
- 功能模块图；
- 核心业务流程图；
- 初步UI原型。

需求评审通过后，原则上不再随意增加大量功能。

## 十、第四阶段：系统设计

系统设计阶段统一形成一份《系统设计说明书》，其中包括概要设计、UI设计和详细设计，不必拆分为多份厚重文档。

### 1. 概要设计

主要包括：

- 系统总体架构；
- 技术选型；
- 功能模块划分；
- 前后端关系；
- 系统运行环境；
- 核心业务流程。

### 2. UI设计

主要包括：

- 页面结构和导航关系；
- 用户端和管理端页面；
- 核心页面原型；
- 表单校验；
- 页面交互和提示方式。

不要求专业级高保真设计，但要求核心页面先设计、后开发，避免开发过程中随意拼凑页面。

### 3. 详细设计

主要包括：

- 数据库E-R图；
- 数据表结构；
- 接口设计；
- 核心类或模块设计；
- 权限设计；
- 异常处理；
- 文件上传等特殊功能设计。

### 4. 设计阶段核心关系

> 需求说明系统“要做什么”；设计说明系统“准备怎么做”；代码体现系统“实际如何实现”；测试验证系统“是否按照要求实现”。
> 

### 5. 阶段成果

学生提交《系统设计说明书》，以及系统架构图、功能模块图、UI原型、E-R图、数据表设计和接口设计。设计评审通过后进入主体开发。

## 十一、第五阶段：工程搭建与AI辅助开发

### 1. 教师提供基础脚手架

教师可以提供Spring Boot后端和Vue前端基础工程，包含统一响应、全局异常、数据库配置、基础页面布局、登录权限基础、文件上传示例、README和Git忽略文件等通用内容。

脚手架只解决通用工程问题，不能提前完成学生项目的大部分业务功能，避免学生只是修改名称和数据表。

### 2. 标准AI开发流程

每个功能按照以下流程实施：

> 选择功能 → 查阅需求文档 → 查阅设计文档 → 拆分任务 → 让AI分析现有代码 → 提出实现方案 → 生成或修改代码 → 人工检查 → 运行测试 → Git提交 → 更新开发记录
> 

### 3. AI代码检查要求

AI生成代码后，学生必须检查：

- 是否符合需求；
- 是否符合数据库和接口设计；
- 是否破坏已有功能；
- 是否重复创建已有模块；
- 是否处理输入校验、权限和异常；
- 是否存在明显安全问题；
- 是否能够真实编译和运行；
- 是否增加了未经批准的功能。

> AI生成代码不等于完成。代码符合文档、能够运行并通过测试，才算真正完成。
> 

### 4. 开发阶段检查节点

| 检查节点 | 主要检查内容 |
| --- | --- |
| 第一次 | 工程能够运行、数据库能够连接、登录或基础权限完成 |
| 第二次 | 核心业务流程基本完成 |
| 第三次 | 主要功能完成，前后端基本联调 |
| 第四次 | 功能完善，进入系统测试 |

学生持续提交Git仓库、阶段版本、开发进度、AI辅助开发记录和问题记录。

## 十二、第六阶段：系统测试与问题修复

### 1. 教师示范内容

教师示范：

- 如何从需求文档提取测试点；
- 如何编写测试用例；
- 如何测试正常流程、错误输入和异常流程；
- 如何进行权限测试；
- 如何记录Bug；
- 如何使用AI分析错误；
- 如何修复问题并进行回归测试。

每一项核心需求都应至少有一个测试用例进行验证。例如，需求规定“未登录用户不能发布信息”，测试中就必须包含未登录访问发布功能的场景。

### 2. 测试范围

课程项目以功能测试为主，包括：

- 核心功能测试；
- 完整业务流程测试；
- 表单输入和边界情况测试；
- 权限测试；
- 异常情况测试；
- 简单兼容性检查；
- 必要的响应速度和运行稳定性检查。

不强制要求复杂压力测试和专业安全测试。

### 3. 阶段成果

学生提交：

- 《系统测试报告》；
- 测试用例表；
- Bug记录表；
- 问题修复记录；
- 测试总结。

## 十三、第七阶段：部署上线与项目答辩

### 1. 部署上线

教师示范项目打包、数据库初始化、配置修改、服务启动、Nginx基础配置和常见部署问题排查。

学生提交：

- 《系统部署说明》；
- 数据库初始化脚本；
- 可运行程序或部署包；
- 系统访问地址；
- 测试账号。

### 2. 项目答辩

学生重点展示：

- 项目背景和用户需求；
- 系统设计；
- 核心业务流程；
- 功能演示；
- 关键技术实现；
- AI辅助开发案例；
- 测试和问题修复；
- 团队成员贡献；
- 项目不足和后续改进。

最终提交完整源代码、项目文档、答辩PPT、演示视频和个人或团队总结。

## 十四、四次正式评审机制

### 1. 立项评审

审核题目是否合适、功能量是否合理、团队分工是否明确，以及项目是否能够在课程周期内完成。

### 2. 需求评审

审核用户是否明确、业务流程是否完整、功能需求是否清楚，以及需求是否可以被测试验证。

### 3. 设计评审

审核系统架构、数据库、UI和接口是否合理，设计是否覆盖需求并能够指导后续开发。

### 4. 验收评审

审核功能是否真实完成，文档、代码和测试是否一致，系统能否部署运行，以及每位成员是否真实参与并能够解释自己的工作。

## 十六、课程需要强化的三个核心观念

### 1. 文档不是为了应付作业

文档用于统一团队认识、控制项目范围、约束AI生成、指导开发实施、提供测试依据和支持后续维护。

### 2. AI不是项目负责人

AI可以辅助分析、生成代码、解释错误、编写测试和整理文档，但项目目标、业务规则、技术决策和最终质量仍由学生负责。

### 3. 最终评价不能只看功能数量

课程评价还应关注：

- 是否遵循开发流程；
- 文档是否能够指导开发；
- 代码是否与设计一致；
- Git过程是否真实；
- 学生是否能够解释AI生成代码；
- 系统是否经过测试；
- 项目是否能够独立部署和演示。

## 十七、课程思路概括

> 本课程采用“双项目、分阶段、文档驱动、AI辅助”的项目化教学模式。课程前期以校园图书管理系统为教师带练项目，由教师提供需求、设计、数据库、接口和测试部署资料，带领学生使用AI编程工具完成工程搭建、功能开发、系统测试和部署运行，使学生初步体验完整的软件开发流程，认识规范文档在AI辅助开发中的重要作用。
> 

> 
> 

> 在带练项目基础上，学生以个人或不超过3人的团队开展自主项目。课程按照项目立项、需求分析、系统设计、工程搭建、迭代开发、系统测试、部署上线和项目答辩等阶段实施，每个阶段均采用“教师示范、学生模仿、成果提交、教师评审、修改完善”的方式推进。
> 

> 
> 

> 课程强调以需求和设计文档约束AI生成过程，要求AI辅助开发必须遵循项目功能范围、数据结构、接口规范、UI设计和编码规范。学生不仅要学会使用AI提高开发效率，还应具备需求理解、方案判断、代码审查、错误调试、系统测试和项目交付能力，最终形成完整、可运行、可部署、可展示的软件项目成果。
